# AI Council Audit — ai-content-pipeline
**Date:** 2026-04-20  
**Auditors:** Claude Sonnet 4.6 (Sonnet lane) · Codex GPT-5.4 (background adversarial) · Semgrep 1.157.0  
**Scope:** All 6 workflow JSON files + README + SETUP  
**Production impact:** YES — 6,800+ executions/month on n8n Cloud

---

## Executive Summary

The pipeline is structurally sound with good engineering practices throughout: proper error routing, retry configuration on every write node, rate-limited alerts, and consistent credential redaction. **One critical security gap was found: the README documents HMAC-SHA256 webhook authentication that is not implemented in the actual workflow.** Three other meaningful issues were identified and fixed in this pass.

---

## Findings

### 🔴 CRITICAL — 1 finding

#### C-1 · Webhook HMAC auth documented but not implemented
**File:** `workflows/05-form-intake.json`  
**Node:** `Webhook`

The Security Architecture table in README states:
> "Webhook authentication — HMAC-SHA256 verification in intake Code node — execution stops immediately on signature mismatch"

**Reality:** The Webhook node has `options: {}` (no rawBody capture). There is no HMAC verification code node anywhere in the workflow. The webhook accepts any POST to `[WEBHOOK_ID]` without validation.

**Impact:** Anyone who discovers or guesses the webhook URL can inject arbitrary form data. This could flood the Leads sheet with fake applications, trigger DocuSign emails to real or fake addresses, and consume API quota.

**Fix applied:** Added `Verify Webhook Signature` Code node immediately after the Webhook node. Enabled `rawBody: true` on the Webhook node options to capture the pre-parsed body for signature verification. Secret stored as n8n variable `$vars.WEBHOOK_SECRET`. See updated `workflows/05-form-intake.json`.

### 🟠 HIGH — 1 finding

#### H-1 · Duplicate applicants receive DocuSign email before dedup check runs
**File:** `workflows/05-form-intake.json`  
**Execution path:** `Append row (Non-startup)` → `Send DocuSign email` → `Get row(s)` → `Check for duplicates` → `If`

The email fires unconditionally immediately after row append. The dedup check runs *after* the email. A second submission from the same email address receives a full DocuSign workflow email before being flagged as a duplicate.

**Impact:** Duplicate applicants receive emails they shouldn't. Support confusion, potential compliance exposure (in regulated lending context, duplicate processing records matter).

**Fix applied:** Reordered the non-startup path: `Append row` → `Get row(s)` → `Check for duplicates` → `If (duplicate?)` → `false branch: Send DocuSign email`. Email is now conditional on the dedup result. Startup rejection path is unchanged (rejections don't need dedup gating).

### 🟡 MEDIUM — 2 findings

#### M-1 · Real n8n credential IDs exposed in 3 workflow files
**Files:** `workflows/02-rss-intake.json`, `workflows/04-content-pipeline.json`

5 internal n8n credential reference IDs were not redacted to `[CREDENTIAL_ID]` placeholder, inconsistent with the rest of the portfolio sanitization:
- `8UYwvprhMGHKF6ra` (Google Sheets, 02-rss-intake)
- `bGDlX6HiAGhitXLG` (OpenAI, 04-content-pipeline)
- `jr25MKfV6H2zSwxO` (Google Sheets, 04-content-pipeline — 4 occurrences)
- `Vdifwq9lKoSXdEIj` (Google Docs, 04-content-pipeline — 2 occurrences)
- `apu6UY3Ke6WNXjpp` (WordPress, 04-content-pipeline)

Note: These are internal n8n instance identifiers, not API keys — they cannot be used to access any external service. The risk is identity leakage about the specific n8n instance structure. Still inconsistent with the portfolio's own redaction standard.

**Fix applied:** All replaced with `[CREDENTIAL_ID]` placeholder.

#### M-2 · `$getWorkflowStaticData` rate limiter resets on n8n Cloud restart
**File:** `workflows/00-error-handler.json`  
**Node:** `Normalize Error Data`

The email rate limiter (`max 3 per workflow per 30 min`) uses `$getWorkflowStaticData('global')`. On n8n Cloud, this in-memory store resets on workflow save, deployment, or n8n Cloud maintenance. During an extended outage that coincides with a Cloud restart, the 30-minute counter can reset mid-incident, allowing another burst of emails.

**Not fixed in this pass** (would require replacing the in-memory counter with a Sheets-backed counter — a Phase 2 item). Added documentation note to README under Lessons Learned and a code comment in the node.

### 🔵 LOW — 3 findings

#### L-1 · `ignoreSSL: true` on RSS feed reader
**File:** `workflows/02-rss-intake.json`  
**Node:** `RSS Read`

SSL verification is disabled. If any RSS feed URL is ever misconfigured to a non-HTTPS endpoint or if a feed host's certificate lapses, content could be silently fetched over plaintext.

**Fix applied:** Removed `ignoreSSL: true`. n8n's default behaviour validates SSL certificates.

#### L-2 · Model ID discrepancy between README and workflow JSON
**Files:** README.md vs `workflows/04-content-pipeline.json`

README badges and Tech Stack table reference `GPT-4.1-mini`. Actual workflow JSON uses `gpt-5.4-mini` (both `Message a model` and `Refine Article` nodes). The README was written when the model was GPT-4.1-mini and not updated after the upgrade.

**Fix applied:** Updated README Tech Stack table and badges.

#### L-3 · `Parse First Pass` in 04 missing `pairedItem` (latent risk, not a current bug)
**File:** `workflows/04-content-pipeline.json`  
**Node:** `Parse First Pass`

`Parse First Pass` is inside the `Loop Over Items: APPROVED` splitInBatches loop and does not declare `pairedItem: { item: 0 }`. At current default batch size of 1, this is not a runtime bug — each loop iteration processes exactly one item so item index resolution is unambiguous. If batch size were ever increased to >1, downstream nodes referencing `$('Parse First Pass').item` could resolve to the wrong loop item.

**Fix applied:** Added `pairedItem: { item: 0 }` to the return object for defensive consistency with the rest of the pipeline's pairedItem pattern.
